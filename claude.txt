# Shorten 프로젝트 개요

## 한 줄 소개

`Shorten`은 드라마/영상 원본을 업로드한 뒤, 분석 파이프라인을 통해 쇼츠 후보를 만들고, 대본 초안·비디오 초안·export 단계까지 이어지는 로컬 중심 쇼츠 편집 보조 MVP입니다.

AWS 같은 관리형 인프라 없이도 동작하도록 설계되어 있으며, 기본 실행 형태는 `FastAPI + Postgres + Redis + Celery + Next.js` 조합입니다. 빠른 스모크 테스트에서는 SQLite와 eager Celery로도 흐름 검증이 가능합니다.

## 프로젝트 목표

- 긴 영상(에피소드)에서 쇼츠로 만들 만한 구간 후보를 자동 생성한다.
- 후보별로 대본 초안과 비디오 초안을 이어서 생성할 수 있게 한다.
- 전체 흐름을 클라우드 의존성 없이 로컬 PC나 단일 서버에서 재현 가능하게 만든다.
- 운영 UI를 통해 업로드, 분석, 진행 상태 확인, 후보 검토, 드래프트/익스포트 확인까지 수행할 수 있게 한다.

## 상위 구조

| 경로 | 역할 |
| --- | --- |
| `backend/` | FastAPI API, SQLAlchemy 모델, Celery 작업, 분석/후보/대본/비디오 초안 서비스 |
| `frontend/` | Next.js App Router 기반 운영 UI |
| `docker-compose.yml` | Postgres, Redis, API 서버, Celery worker 실행 |
| `plan.md` | 긴 설계 메모 및 작업 계획 |
| `claude.md` | 이 프로젝트를 빠르게 이해하기 위한 요약 문서 |

## 아키텍처 요약

이 프로젝트는 프런트엔드와 백엔드가 명확히 분리된 구조입니다.

- 프런트엔드는 `frontend/`의 Next.js App Router 앱이며, 운영자가 브라우저에서 에피소드를 업로드하고 분석 상태와 결과를 확인하는 UI를 제공합니다.
- 백엔드는 `backend/app/main.py`를 진입점으로 하는 FastAPI 서버이며, `/api/v1` 아래에 에피소드, 잡, 후보, 스크립트 초안, 비디오 초안, export 관련 REST API를 제공합니다.
- 무거운 처리나 순차 파이프라인은 Celery task로 분리되어 Redis를 broker/result backend로 사용합니다.
- 메타데이터는 DB(Postgres 또는 SQLite)에 저장하고, 실제 업로드 영상/중간 산출물/렌더 파일은 로컬 디스크에 저장합니다.

즉, "메타데이터는 DB", "비동기 작업은 Celery", "실제 파일은 로컬 스토리지", "운영 화면은 Next.js"라는 역할 분리가 핵심입니다.

## 백엔드 구조

### 진입점

- `backend/app/main.py`
  - FastAPI 앱 생성
  - CORS 설정
  - `/health` 제공
  - API 라우터를 `/api/v1` prefix로 mount
  - SQLite일 때만 `create_all` 수행

### 라우터 구성

`backend/app/api/v1/router.py`에서 다음 도메인 라우터를 묶습니다.

- `episodes`
- `jobs`
- `candidates`
- `script_drafts`
- `video_drafts`
- `exports`

즉, 백엔드는 "에피소드 업로드/분석 시작 -> 잡 추적 -> 후보 조회 -> 스크립트 초안 -> 비디오 초안 -> export" 흐름을 REST API로 제공합니다.

### 핵심 서비스 계층

`backend/app/services/`는 도메인 로직의 중심입니다.

- `analysis_service.py`
  - 업로드된 에피소드 ingest
  - 프록시 트랜스코딩
  - 샷 탐지
  - 키프레임 추출
  - 자막 추출/생성
  - 신호 계산
  - 후보 생성
- `candidate_generation.py`
  - 쇼츠 후보 생성 로직
- `vision_candidate_refinement.py`
  - 비전/LLM 기반 후보 재정렬 보조
- `script_service.py`
  - 후보 기반 스크립트 초안 생성
- `video_draft_service.py`
  - 비디오 초안, rerender, export 관련 처리
- `short_clip_service.py`
  - FFmpeg 기반 숏클립 렌더링
- `storage_service.py`
  - 로컬 파일 저장 경로 및 스토리지 레이아웃 관리
- `jobs.py`
  - 비동기 job 상태 전이와 진행률 관리

### 비동기 파이프라인

`backend/app/tasks/pipelines.py`에서 Celery task를 정의합니다.

분석 파이프라인은 대략 아래 순서로 연결됩니다.

1. `ingest_episode`
2. `transcode_proxy`
3. `detect_shots`
4. `extract_keyframes`
5. `extract_or_generate_transcript`
6. `compute_signals`
7. `generate_candidates`

이후 별도의 작업으로 다음이 이어질 수 있습니다.

- 스크립트 초안 생성
- 숏클립 렌더링
- 비디오 드래프트 생성/재렌더
- export 생성

즉, 분석과 생성 작업은 HTTP 요청 하나로 끝나는 구조가 아니라, API가 job을 만들고 Celery worker가 실제 처리를 수행하는 구조입니다.

## 프런트엔드 구조

### 라우팅 방식

프런트는 `frontend/app/` 기반의 App Router 구조입니다.

- `/` -> `/episodes` 리다이렉트
- `/episodes`
- `/episodes/new`
- `/episodes/[episodeId]`
- `/episodes/[episodeId]/candidates`
- `/candidates/[candidateId]`
- `/drafts/[draftId]`
- `/exports/[exportId]`

### UI 역할

프런트는 백엔드 API를 직접 호출하는 운영 UI입니다.

- 에피소드 목록 조회
- 영상 업로드
- 분석 시작 / 재분석 / 캐시 정리
- 잡 진행 상태 폴링
- 타임라인/후보 목록 확인
- 후보 상세 확인
- 비디오 초안 및 export 결과 확인

### 데이터 패칭 전략

- 서버 컴포넌트에서 초기 데이터 로드
- 클라이언트 컴포넌트에서는 React Query로 job 상태 폴링
- API 호출은 `frontend/lib/api.ts`, `frontend/lib/public-api.ts`에 모아둠
- 기본 API endpoint는 `NEXT_PUBLIC_API_BASE_URL` 환경 변수 사용

즉, "초기 렌더는 서버 컴포넌트", "실시간 상태는 React Query 폴링" 전략을 사용합니다.

## 데이터 흐름

가장 중요한 사용자 흐름은 다음과 같습니다.

1. 운영자가 에피소드 영상을 업로드한다.
2. 백엔드는 원본 파일을 로컬 스토리지에 저장하고 DB에 에피소드 레코드를 만든다.
3. 분석 API를 호출하면 job이 생성되고 Celery 파이프라인이 실행된다.
4. 파이프라인이 샷, 자막, 키프레임, 신호를 계산한 뒤 쇼츠 후보를 생성한다.
5. 프런트는 jobs API를 폴링해 진행 상태를 보여준다.
6. 운영자는 후보 목록과 후보 상세를 확인한다.
7. 선택한 후보에 대해 스크립트 초안과 비디오 초안을 생성한다.
8. 최종적으로 export 정보를 확인한다.

이 흐름은 "업로드 -> 분석 -> 후보 탐색 -> 드래프트 생성 -> export"로 이해하면 됩니다.

## 현재 프로젝트 성격 정리

이 프로젝트는 완성형 소비자 서비스라기보다, 영상 분석과 쇼츠 제작 파이프라인을 하나의 로컬 실행 가능한 제품 형태로 묶어 둔 운영형 MVP에 가깝습니다.

핵심 가치는 다음 세 가지입니다.

- 클라우드 의존성을 최소화한 로컬 중심 실행
- 영상 분석부터 후보/대본/드래프트/export까지 이어지는 end-to-end 흐름
- 백엔드 파이프라인과 프런트 운영 UI가 연결된 실사용형 구조
